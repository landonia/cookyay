## Task
v5 runtime auto-block needs the v4 signature data available *in the browser*, but
the scanner DB carries scanner-only fields (cookies, localStorage, source
provenance) that are useless for blocking scripts/iframes and would waste bundle
budget [goals.md §What ships in v5]. Add a codegen step that emits a stripped,
client-safe slice of `data/services.yaml` — id, category, requestHosts,
requestPaths only — as `packages/cookyay/src/db-autoblock.generated.ts`, mirroring
the proven `build-services-db.mjs` → `db-curated.generated.ts` pipeline. This is
the foundation every other v5 task builds on. Research settled that inlining this
slice (vs. a lazily-loaded asset) is correct — a fetched DB arrives too late to
block scripts that execute within milliseconds [research/performance-engineer.md].

## Implementation notes
- Extend `packages/scanner/scripts/build-services-db.mjs` with a second emitter (or
  add a sibling script wired into the cookyay package prebuild) that writes the
  client module. Keep generated files committed, as `db-curated.generated.ts` is.
- Mark Google-owned services with a flag (or a derivable owner field) so task 002's
  matcher can skip them — the CM v2 skip-Google decision [research/_index.md §Update].
- Author/confirm `scriptUrlGlobs` in `services.yaml` for any service sitting on a
  shared CDN host, so the runtime matcher can disambiguate (single-host match would
  false-positive). This is light data-authoring within this task.
- Reuse the existing host/path field shapes from `db.ts` so scanner and client stay
  in lockstep (task 007 asserts parity).

## Out of scope
- The runtime matcher logic itself (task 002).
- The `autoBlock` config flag and tree-shaking wiring (task 003).
- Any interception/blocking behavior (tasks 004, 005).

- `packages/cookyay/src/db-autoblock.generated.ts` — New generated file: 50-service client-safe slice, 8.1KB raw / 1.8KB gzip, typed against `AutoBlockEntry`.

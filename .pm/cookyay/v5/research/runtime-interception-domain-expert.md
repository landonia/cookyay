# Runtime-interception domain-expert — Research findings

## Summary

- The only reliably race-free mechanism for blocking third-party `<script src>` tags that arrive in the **initial HTML** is a synchronous `document.createElement`/`appendChild` override installed by the bootstrap snippet before any other parsing — `MutationObserver` is too late for parser-inserted nodes and cannot be relied upon alone.
- `<iframe>` and pixel/beacon requests (`facebook.com/tr`) are categorically different from scripts: iframes can be intercepted via the same DOM API override, but 1x1 `<img>` pixels fired from inline `<script>` code cannot be stopped without blocking the parent script first.
- Runtime auto-block and Consent Mode v2 must be carefully layered: the banner must never both block a script **and** fire a `gtag('consent', 'update', 'denied')` for the same service — that is double-handling and can suppress Google's own graceful degradation path [prd.md §3.4].
- The <20KB budget [prd.md §3.1 / goals.md §Honor the <20KB budget] is the hard constraint on signature-DB delivery: the full `services.yaml` (~50 entries with globs) must be compiled to a minimal host/path lookup structure, not shipped as raw YAML or with scanner-side classifier logic.

## Findings

1. **`document.createElement` override is the only sync-safe approach for initial-HTML scripts** [goals.md §Client-side signature recognition, prd.md §5 two-part bootstrap]. The browser's HTML parser fetches and executes `<script src>` as soon as it encounters the tag. A `MutationObserver` callback fires asynchronously (microtask/macrotask boundary), meaning the browser has already dispatched the network request — and for same-origin or cached scripts may have already begun execution — before the observer runs. Overriding `document.createElement('script')` in the synchronous bootstrap snippet intercepts all programmatic script creation. For parser-inserted nodes, the companion technique is overriding `Element.prototype.setAttribute` to intercept the moment `src` is set (the parser sets `src` last, triggering the fetch). Both overrides must be installed inside the `<1KB synchronous inline snippet` [prd.md §5], before GTM/GA tags appear in markup.

2. **`beforescriptexecute` is dead — Firefox-only and removed in Firefox 59** [goals.md §What ships in v5]. It cannot be used. Service Workers are theoretically capable of intercepting all fetches but require a prior page load (first-visit gap), a `ServiceWorkerRegistration` scope that matches the third-party origin (impossible for cross-origin scripts), and add significant setup cost that violates the `<20KB / plain <script> tag` constraint [prd.md §5]. CSP `script-src` blocking would require the site to dynamically inject a response header or meta-tag CSP *before* parsing begins, which is only feasible server-side and is not something the banner JavaScript can control.

3. **The race-condition window for scripts already in the initial HTML is real and bounded** [prd.md §5, goals.md §Acceptance bar]. If a site pastes `<script src="https://www.googletagmanager.com/gtm.js?id=GTM-…">` *before* the cookyay bootstrap snippet, the script is already fetched and executing before any interception is possible. The mitigation is documentation posture: the bootstrap must be the **absolute first `<script>` in `<head>`** [architecture.md §1], and the acceptance bar test fixture must enforce this. Scripts placed after the bootstrap are blockable via the `createElement`/`setAttribute` override. For the dogfooding acceptance bar [goals.md §Acceptance bar], the live site must have the bootstrap first — if it does not, auto-block will silently have no effect for scripts present in initial HTML.

4. **iframes can be blocked by the same `createElement` override** [goals.md §Client-side signature recognition]. When the `createElement('iframe')` factory is wrapped, the interceptor inspects the eventual `src` attribute (again, set last by the parser or by JS). An iframe without `src` (e.g. set via JS after creation) is caught by wrapping `HTMLIFrameElement.prototype` setter for `src`. The interceptor replaces the live `src` with a data URI blank or removes it, inserts a placeholder `<div>` (reusing the existing `_buildPlaceholder` pattern in `blocking.ts`), and queues the element for activation on consent grant.

5. **1x1 pixel beacons (`<img src="https://facebook.com/tr?…">`) cannot be intercepted at the network level by banner JS alone** [goals.md §Client-side signature recognition]. These fire from within third-party script code; blocking the parent script (GA, Meta Pixel loader) is the correct chokepoint. Wrapping `HTMLImageElement` `src` setter would catch dynamically created pixels but adds significant surface area and will break lazy-loaded first-party images if the matcher has false positives. Recommendation: block at script level only; surface a warning in the auto-block report for known pixel hosts where the parent script was not intercepted.

6. **Declared rules must be checked before auto-block to prevent double-registration** [goals.md §Declared rules always win over auto-detected ones, prd.md §3.2]. The existing `blocking.ts` `scanBlocked()` already sets `data-cookyay-state="blocked"` on declared elements. The auto-block path must check for this attribute before registering an element, exactly as `_registerScript` already guards with `if (el.getAttribute(ATTR_STATE) === STATE_BLOCKED) return`. The lookup order must be: (1) check `data-cookyay-state` attribute — if already blocked, skip; (2) check declared config rules; (3) only then attempt signature match. This also means declared `type="text/plain"` scripts are fully invisible to the auto-block interceptor — the `createElement` override is only triggered by live/active script nodes, not inert ones.

7. **Consent Mode v2 interaction: auto-block must not double-deny** [prd.md §3.3, §3.4]. The bootstrap snippet already fires `gtag('consent', 'default', {…all denied…})` [bootstrap.ts line 120]. Google's own tags (GTM, GA4) honor this and do not fire analytics until `gtag('consent', 'update', 'granted')`. If auto-block *also* intercepts and removes the GTM `<script>` tag, GTM never loads at all — `update` signals from the banner will fire into a void dataLayer with no listener. The correct behavior: when auto-block is enabled for an analytics/marketing service AND Consent Mode v2 is active, the interceptor should still allow Google's own tags to load (they degrade gracefully under CM v2 denied defaults), and block only the data collection that happens outside CM v2's own gating. Alternatively, document this as an explicit opt-out: `autoBlock: { skipConsentModeServices: ['ga4', 'gtm'] }`.

8. **Bundle budget for signature DB delivery** [goals.md §Honor the <20KB budget, architecture.md §2]. The `services.yaml` ~50 entries include cookies, localStorage keys, requestHosts, requestPaths, scriptUrlGlobs, iframeSrcGlobs. At runtime, only `requestHosts`, `requestPaths`, `scriptUrlGlobs`, and `iframeSrcGlobs` matter for DOM interception — cookies and localStorage keys are observable after the fact, not during load. A client-only compiled lookup structure (flat array of `{id, category, hosts: string[], paths: string[], scriptGlobs: string[]}`) for 50 services, without any scanner-side classifier logic, should compress to ~2–3KB gzipped — comfortably within budget if added to the deferred UI bundle rather than the bootstrap snippet.

9. **Withdrawal/re-grant and auto-block** [prd.md §3.2]. The existing withdrawal posture (reload recommended, `showWithdrawalPrompt()` in `withdrawal.ts`) applies equally to auto-blocked scripts that were subsequently granted and executed. Re-block on withdrawal is impossible without a reload because the third-party script has already run and may have set cookies, attached event listeners, and sent requests. The existing "reload required" toast is the correct and honest mechanism. On consent re-grant after withdrawal (user reloads, grants same category again), the auto-block interceptor must be re-armed at init time — the `createElement` override must check live consent state each time it fires, not only at registration.

## Gotchas

- **Prototype freeze / integrity checks.** Some browser extensions (uBlock Origin, Privacy Badger) also wrap `createElement`. If both run in the same page context, the wrappers chain — order matters. The cookyay wrapper must call through to the original (`const _orig = document.createElement.bind(document); document.createElement = (tag) => { … return _orig(tag) }`) and must not throw on unexpected tag types, or it will break every element creation on the page.
- **Trusted Types CSP.** Sites using Trusted Types (`require-trusted-types-for 'script'`) validate script `src` and `textContent` before execution. Dynamically rewriting a `<script>` node's `src` attribute to block it may itself trip a Trusted Types violation if the browser has already queued the fetch. The auto-block interceptor must intercept *before* `src` is set (during `createElement`) rather than after, to stay Trusted-Types-clean.
- **`document.write` injection.** Some legacy tags (DoubleClick, old AdSense) use `document.write('<scr'+'ipt src=…>')`. Overriding `document.write` is possible but extremely risky — it is used by the browser for streaming HTML parsing, and wrapping it can break page rendering. Scope `document.write` interception to after DOMContentLoaded if attempted at all, and treat it as best-effort rather than guaranteed.
- **False positives on shared CDNs.** `cdn.jsdelivr.net`, `unpkg.com`, `cdnjs.cloudflare.com` host both tracker scripts and legitimate dependencies. A signature that matches on hostname only (e.g. `jsdelivr.net`) will false-block all CDN-delivered packages. The `scriptUrlGlobs` field in `services.yaml` must require path-level specificity for shared CDN hosts — v4's `requestPaths` model is the right pattern [architecture.md §Amendment 2026-06-10].
- **GTM-injected tags.** GTM dynamically injects `<script>` tags via `document.createElement` after its own container loads. If GTM itself is not blocked (see Finding 7), its injected tags will be caught by the `createElement` override. This is actually the desired behavior for marketing tags injected by GTM — but it requires the signature matcher to run on GTM-injected URLs too, not just on static markup.

## Recommendations

1. **Install the `document.createElement` + `Element.prototype.setAttribute` (for `src`) override in the synchronous bootstrap snippet** — this is the only race-free path for blocking scripts placed after the snippet in `<head>`. Keep it under 300 bytes minified (it only needs to: check the tag type, extract the eventual `src`, match against a pre-loaded signature list, and either chain through or queue the element).
2. **Compile the signature DB to a client lookup module** — a generated `src/auto-block-db.generated.ts` containing only the fields needed at DOM-intercept time (`id`, `category`, `hosts`, `paths`, `scriptGlobs`, `iframeSrcGlobs`), built from `services.yaml` by the same codegen pipeline as `db-curated.generated.ts`. This module ships in the deferred UI bundle, not the bootstrap snippet.
3. **Default posture: block high-confidence matches only** (services with at least one `scriptUrlGlob` or `iframeSrcGlob`, or a `requestHost` exclusive enough not to be a shared CDN). Medium-confidence host-only matches should warn but not block by default — configurable via `autoBlock: { minimumConfidence: 'high' | 'medium' }`.
4. **Explicitly exclude or opt-through known Consent Mode v2 services** (GA4, GTM) when CM v2 is active, or document the behavior clearly. Do not auto-block CM v2 tags that are already degrading gracefully under denied defaults.
5. **Reuse the existing `_registerScript` / `_registerIframe` queuing infrastructure** in `blocking.ts` for auto-detected elements — add an `autoDetected: true` flag to `QueueEntry` for observability/debugging, but otherwise share the grant/inject path entirely to avoid divergence.
6. **Instrument with `debug: true` logging** [architecture.md §8]: every auto-block intercept should log `[Cookyay] auto-blocked <script src="…"> (service: ga4, category: analytics)` when debug mode is on, making it easy to diagnose false positives.

## Open questions for the user

1. **Consent Mode v2 pass-through policy**: should auto-block skip Google tags entirely (letting CM v2 do the degradation), or should the user be able to configure blocking of GTM/GA4 at the DOM level even under CM v2? This has product-compliance implications — blocking GTM prevents all CM v2 `update` signals from firing.
2. **`document.write` scope**: is legacy `document.write`-based ad injection (DoubleClick, old AdSense) in scope for v5, or explicitly deferred given the page-breakage risk?
3. **Shared-CDN false-positive policy**: should a `scriptUrlGlob` be required for any service whose `requestHosts` entry is a shared CDN hostname? Or is CDN-delivered blocking handled by the `minimumConfidence` gate at runtime?
4. **Pixel/beacon interception**: is `<img>`-based pixel blocking (Meta Pixel `facebook.com/tr`, Google Floodlight) explicitly in or out of scope for v5? It requires wrapping `HTMLImageElement.src` setter, which is higher breakage risk than script interception.
5. **First-party script paths** (e.g. a site that self-hosts GA4 at `/js/ga.js`): the signature DB has no coverage here. Should v5 document this gap, or add a self-hosted-path escape hatch to the config?

## Out of scope

- Runtime heuristics for services NOT in the curated DB (~50 entries) — explicitly deferred to a later version [goals.md §What's deferred to later versions].
- Server-side CSP injection or network-layer blocking (Service Workers, proxy middleware) — requires infrastructure the banner cannot control [prd.md §5 constraints].
- `beforescriptexecute` (removed from all browsers) and any Firefox-specific interception.
- Automatic cookie deletion on withdrawal — already out of scope; `clearOnWithdraw` callback is the mechanism [config.ts `CookyayConfig.clearOnWithdraw`].
- Materially expanding the DB beyond ~50 services — community-driven, ongoing, not a v5 deliverable [goals.md §What's deferred to later versions].

## Task
Correct the five concrete false-positive signatures the tracker SME found in the
existing curated set [research/domain-expert-trackers.md ¬ßSummary]. These produce
spurious blocking declarations that would erode trust in the scanner's output
[prd.md ¬ß3.6]. Edit the signatures in `services.yaml` (from 001) and lock each fix
with a regression test.

## Implementation notes
- The five fixes: (1) `t.co` ‚Üí drop/narrow for Twitter Pixel; (2) `facebook.com`
  ‚Üí `connect.facebook.net` + `facebook.com/tr` path (path support lands in 004 ‚Äî
  if 004 isn't merged yet, narrow to the host and add the path signal once
  available); (3) Vimeo generic `player` cookie ‚Üí use a vimeo-specific signal;
  (4) Mixpanel `mp_` ‚Üí require a requestHost cross-check, not cookie-alone;
  (5) `_ga` shared across GA4/UA/GTM ‚Üí attribute by the co-fired requestHost.
- Several of these are most cleanly expressed with `requestPaths` (004) and the
  two-signal confidence rule (006); coordinate so the fix isn't undone. Where a
  fix needs path matching, it's acceptable to land the data here and let 004 make
  the matcher honor it ‚Äî note the dependency in the PR.

## Out of scope
- Adding new services (005). General confidence-model rework (006) ‚Äî only the
  `mp_`/`_ga` cross-check needed to remove these specific false positives.

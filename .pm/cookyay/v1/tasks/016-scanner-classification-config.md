## Task
Turn raw crawl findings into the deliverable that makes the scanner sticky: a ready-to-use config with honest confidence annotations. Classification rides on the Open Cookie Database + a curated top-20 — bounded scope, community-extensible (prd §7 risk mitigation).

## Implementation notes
- Check Open Cookie Database license compatibility with Apache-2.0 and attribute properly.
- The fingerprint fixture JSON (012) keeps unit and E2E layers agreeing on what "detected" means.
- Confidence: high (exact known cookie name + host), medium (host match only), low (heuristic).

## Out of scope
Auto-updating the database at scanner runtime (build-time ingest only for v1), classifying every artifact (unclassified bucket is expected).

## Verifier notes — 2026-06-07 — REJECTED
**Verifier:** Senior QA / Tech Lead
**Summary:** The round-trip E2E criterion is not implemented (the emitted config is never applied to the fixture site), the "OCD ingested at build time" claim is overstated (a ~21-entry hand-transcribed subset, no ingest mechanism), and `findServiceByHost` has a substring-match bug that misclassifies arbitrary hosts as X (Twitter) Pixel.

**What needs to change:**
1. **Criterion 5 (round-trip) is not actually tested.** `packages/scanner/e2e/scanner-classify.spec.ts:70–116` only asserts the emitted config *contains* analytics/marketing entries; nothing ever applies the config to the fixture site and verifies the synthetic trackers are blocked. The criterion reads "produces a config that, applied to the fixture site, blocks the synthetic trackers (round-trip E2E test)". The Implementation summary quietly reworded this to "correctly identifies analytics + marketing categories". Fix: add an E2E test (or a small harness page under `fixtures/`) that calls `Cookyay.init(<emitted config>)` on the fixture (e.g. a variant of `fixtures/blocking/all.html` that accepts an injected config instead of its inline one), then asserts pre-consent the stubs stay inert (`window.__ga4Ran`/`__pixelRan` undefined, iframe has no `src`) and post-grant they execute — same assertions `e2e/blocking.spec.ts` already uses. This also proves the "ready-to-use" claim in criterion 2 (e.g. that the extra `_meta`/`_unclassified`/`_noscriptWarnings`/`_scanMeta` keys don't trip `validateConfig` in `packages/cookyay/src/config.ts`).
2. **`findServiceByHost` substring match produces false positives** — `packages/scanner/src/db.ts:439`: `host.includes(h)` after the exact and `.endsWith('.'+h)` checks is purely a false-positive generator. With `'t.co'` in the X (Twitter) Pixel `requestHosts`, ANY host containing "t.co" (verified: `giant.com`, `audit.com`, `react.com`) is classified as X Pixel / marketing / medium confidence. This directly undermines the "honest confidence annotations" deliverable (task body) and the goals.md acceptance bar ("correctly classified common services" on a real site). Fix: drop the `includes` branch; keep exact match + `endsWith('.'+h)` only, and remove bare `t.co` or require full-host matching for it.
3. **Criterion 1 "Open Cookie Database (ingested at build time)" is overstated.** `db.ts:316–384` is ~21 hand-transcribed entries *labeled* `source: 'ocd'`; there is no ingest of the OCD CSV anywhere (no build script, no generated artifact). Research `domain-expert-cmp.md §Recommendations 3` says "Ingest the CSV at scanner build time" precisely to get broad coverage and reduce maintenance burden — neither benefit exists with a hand-typed subset, and no deviation is documented (the summary claims ingestion happened). Fix: either (a) add a small build-time ingest script (e.g. `scripts/ingest-ocd.mjs` that reads the OCD CSV, maps its categories to the four cookyay categories, and emits a generated `db-ocd.generated.ts` checked in or built in `prebuild`), or (b) document an explicit, justified deviation in the Implementation summary AND correct the attribution/claims in `db.ts`/summary to say "curated subset derived from OCD".
4. Minor (fix while in there, not individually rejectable): (a) `config-emitter.ts:198–200` confidence-upgrade branch is dead code — `findServiceByHost` only ever returns `'medium'`; (b) `config-emitter.ts:224/251` silently drops scripts/iframes whose effective category is `necessary` from the output entirely — at minimum surface them somewhere; (c) confidence semantics deviate from the impl notes ("high = exact cookie name + host" — implemented as "high = any curated cookie match", no host cross-check); the deviation is documented in `db.ts:14–17` so just confirm it's intentional in the summary.

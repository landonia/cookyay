What Cookyay cannot do

Honesty is the point. Here is what Cookyay deliberately does not do, and why. If any of these are hard requirements for your site, you should use a different tool.

  • No server-side consent log. Cookyay stores consent in the visitor's browser only (cookyay_consent cookie + localStorage). If a regulator asks "show me the consent record for user X on date Y," Cookyay gives you no way to answer — the data lives only in that user's browser and is inaccessible to you. For full GDPR Art. 7(1) accountability, you must forward consent events to your own backend. The cookyay:consent CustomEvent payload includes timestamp, policy version, and per-category choices — enough to build an audit trail if you handle the forwarding. See Compliance limitations for the forwarding snippet. CookieYes (paid) hosts this log for you.
  • No IAB TCF certification. Cookyay is not a certified Transparency and Consent Framework CMP. If your site surfaces IAB-registered vendor lists or relies on TCF consent strings (common for adtech and programmatic advertising networks), Cookyay is the wrong tool. TCF requires ongoing vendor list management, consent string encoding, and registration with IAB Europe — well outside the scope of a lean self-hosted library.
  • No geo-detection. Cookyay shows the opt-in consent banner to every visitor, everywhere — including US visitors who may not be subject to GDPR. This is a deliberate design choice (the "strictest everywhere" posture) that simplifies implementation and avoids IP geolocation infrastructure. If you need to show a lighter experience (or no banner at all) to non-EU/CA visitors, Cookyay cannot do this.
  • No hosted dashboard. There is no web UI showing consent rates, banner performance, or site analytics. Cookyay ships no infrastructure — you are the operator. If you need visibility into how many visitors accepted or declined, you must build that from the forwarded consent events.
  • No automatic script detection. In v1, Cookyay uses declarative blocking only: you must add type="text/plain" data-category="..." to every script tag you want blocked. The @cookyay/scanner CLI helps you find what to declare, but the HTML changes are manual. CookieYes can auto-block known third-party trackers without touching your HTML.
  • No built-in translations. Cookyay ships English text by default. Translated strings must be supplied by the site owner via the strings config object. There are no bundled locale files and no automatic browser-language detection. CookieYes auto-detects the visitor's browser language and serves a translated banner.
  • No guarantee of legal compliance. Cookyay implements the technical consent UX patterns described by GDPR, CCPA, and related regulations. Whether your specific deployment is compliant depends on your jurisdiction, your site's data flows, and how you configure the library. This is not legal advice.

Open-source neighbors

Cookyay is not the only open-source option. Two libraries cover significant overlap and are worth knowing:

Library What it covers What it lacks vs Cookyay Best fit
vanilla-cookieconsent v3
(orestbida) 		Banner UI, opt-in/opt-out per category, all Consent Mode v2 signals (7 signals, superset of the 4 mandatory), onConsent/onChange callbacks, declarative script blocking, 20+ bundled locales, ARIA-accessible. ~24 KB min+gzip. No GPC signal detection or honoring. No CLI scanner. No two-part bootstrap (Consent Mode defaults ordering is user's responsibility). Slightly over the 20 KB budget Cookyay targets. Sites that need built-in multi-language support and are comfortable handling Consent Mode load ordering themselves. The most mature and widely deployed open-source option.
Klaro! Per-service script blocking, i18n with bundled translations, declarative config, actively maintained. No Google Consent Mode v2 support. No GPC honoring. No CLI scanner. No two-part bootstrap. Config schema is more verbose than Cookyay's. Sites that need granular per-service blocking with rich translations but don't rely on Google Analytics/Ads (where Consent Mode v2 matters for revenue signal quality).

If you only need banner UI + Consent Mode v2 and don't care about GPC or a local scanner, vanilla-cookieconsent v3 is the more mature choice — it has a larger user base, more themes, and built-in translations. Cookyay's differentiators are: the two-part bootstrap (Consent Mode ordering handled automatically), GPC detection and honoring (unique among these three as of June 2026), the local CLI scanner (runs against localhost and authenticated pages), and the strictest-everywhere posture as an explicit design principle rather than an option.

When to switch to Cookyay — and when not to

You can likely drop CookieYes and use Cookyay if:

  • You are a developer comfortable editing HTML and running CLI tools.
  • You can manually declare each tracked script in your HTML (or run the scanner to find them).
  • You do not need a hosted consent log — either because your risk profile is low (small site, no DPO) or because you are willing to forward consent events to your own backend.
  • You want full control over the script that runs on your site (self-hosted, auditable).
  • You want GPC to be honored automatically (Cookyay does this; CookieYes does not as of June 2026).
  • You need the scanner to run against a local dev server or authenticated pages.
  • Cost is a concern (Cookyay is free with no per-domain or per-pageview limits).

You should stay on CookieYes (or a similar SaaS CMP) if:

  • You need a server-side consent audit trail. If you are subject to GDPR enforcement risk and need to demonstrate consent records to a regulator on demand, CookieYes's hosted log is the practical solution. Cookyay requires you to build and host this yourself.
  • You need geo-targeted banner variants. Cookyay shows the opt-in banner to everyone. If showing a lighter or no banner to US visitors is important to your UX or business (e.g., SaaS with mostly North American traffic), you need geo-detection that Cookyay does not have.
  • You need auto-detection of third-party scripts. If you cannot edit your HTML to add type="text/plain" attributes (e.g., scripts injected by a CMS you don't control), Cookyay cannot block them in v1.
  • You need IAB TCF compliance. Cookyay is not a TCF-certified CMP.
  • You need built-in translations without configuration. Non-English sites where you cannot supply translated strings need a tool with bundled locales.
  • You manage many domains from a single dashboard. Cookyay has no management UI; each domain is an independent deployment.
Bottom line Cookyay is a developer tool, not a SaaS replacement. It covers the technical consent mechanics — banner, blocking, Consent Mode v2, GPC — at zero marginal cost, with full source transparency. The trade-off is that you own the infrastructure decisions (consent log, multi-domain management, translations) that CookieYes handles for you.
